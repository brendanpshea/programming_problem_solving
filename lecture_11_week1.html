PART 1.5

Core Theory


This is a CS class, not just a build-along. The tools have concepts behind them.

What an IDE Actually Is

IDE = Integrated Development Environment

An IDE brings editing, compiling, running, debugging, navigation, and source control into one coordinated environment.

Why this matters

It reduces friction. Fewer tool jumps means faster feedback, fewer mistakes, and easier experimentation.

1
Edit code in one place
2
Run and see output immediately
3
Debug errors without leaving the environment
4
Version your changes with Git in the same interface

What Git Actually Tracks

ConceptMeaningWhy students care
RepositoryA project plus its version historyYour working folder and its timeline live together
CommitA named snapshot of the project stateYou can return to known-good versions
BranchAn alternate line of developmentUseful later for experiments, but not required yet
RemoteA copy of the repo on another machine/serverGitHub is the remote copy you submit
Git does not care about “projects” in the human sense. It tracks file changes and snapshots over time.

LLMs and Agents Are Not the Same Thing

TermDefinitionCourse meaning
LLM A large language model that predicts likely next text based on context The engine producing code suggestions and explanations
Inline AI Short local completions while you type Good for small edits and boilerplate
Agentic AI A broader workflow where the model follows a scoped request, iterates, and helps you refine output Good for whole features, architecture, debugging, and design conversations
An LLM produces plausible output. It does not know whether your program is correct until you test it.

Target behavior: treat AI output as draft 1, then test and iterate.

PART 4

GitHub & Version Control


A time machine for your code

Why This Matters Right Now

Without version control...

  • AI gives you something broken
  • You try to fix it — now it's worse
  • No way back to what worked
  • SnakeGame_FINAL_v3b.java 😬

With Git + GitHub...

  • Every commit is a named snapshot
  • AI breaks something? Roll back in seconds
  • Work history always visible
  • One clean folder, permanent record
Git is not a backup tool — it's a time machine. GitHub is where the time machine lives in the cloud — and doubles as your portfolio.
Use the vivid scenario: "Imagine Snake is moving correctly. You add food logic. Now nothing works and you don't remember what you changed. With git, one command gets you back to the working version. Without git, you're staring at broken code for an hour."

The Git Mental Model

graph LR A["Edit files on your machine"] -->|"git add ."| B["Staging area (changes queued)"] B -->|"git commit -m '...'"| C["Local repository (snapshot saved)"] C -->|git push| D["GitHub remote repository"] D -->|git clone / git pull| A style A fill:#1e3a5f,stroke:#4299e1,color:#fff style B fill:#1a3a2e,stroke:#48c78e,color:#fff style C fill:#2d2010,stroke:#ffc107,color:#fff style D fill:#2d1020,stroke:#e040fb,color:#fff
You only clone once (to get the repo onto your machine). After that, the loop is: edit → add → commit → push, repeated.
Walk through each node. Students often confuse "commit" (local) with "push" (GitHub). Clarify: a commit creates a snapshot on your machine. Push sends it to GitHub. You can commit many times before pushing.

The Three Commands You'll Use 90% of the Time

# Stage all changed and new files
$ git add .

# Create a snapshot with a descriptive message
$ git commit -m "Add food spawning to SnakeGame"

# Send your commits up to GitHub
$ git push

❌ git commit -m "fixed stuff"
Useless when you look back in two weeks
✅ git commit -m "Add wall collision — game over on edge hit"
Tells you exactly what changed and why
Demo this live: make a trivial edit to README, then run all three commands in the VS Code terminal. Refresh GitHub in the browser. Students love seeing the commit appear live. Also show: VS Code's Source Control tab does all three steps with a GUI if they prefer.

Inline AI Exercise: Small Local Change


import java.awt.*;
import javax.swing.*;

public class SnakeStarter extends JPanel {
    private final int gridSize = 20;
    private final int cellSize = 30;

    @Override
    protected void paintComponent(Graphics g) {
        super.paintComponent(g);
        g.setColor(new Color(40, 40, 40));
        g.fillRect(0, 0, gridSize * cellSize, gridSize * cellSize);
    }
}
Use inline completion to add: "draw a 3-segment green snake near center".

"In SnakeStarter.java, add a main method that opens a 600x600 JFrame titled 'Snake Starter'. Keep one file. Add LinkedList<Point> with 3 segments and draw it in green. Do not add movement yet."

  • Multiple constraints in one instruction

Validation checklist

  • Prompt constraints respected

PART 7

First Look at Agentic AI


Specificity is the skill

Weak vs. Strong Prompts

❌ Weak

"Write me a Snake game in Java"

What will the AI decide for you?

  • Which library? (Swing? JavaFX? AWT?)
  • Window dimensions?
  • Class names and structure?
  • One file or many?
  • Grid size? Speed? Colors?

✅ Strong

"I'm building a Snake game in Java using Swing. Create a single file called SnakeGame.java. It should have a main method that opens a JFrame window (600×600 pixels) with the title 'Snake'. Inside the frame, add a JPanel subclass called GamePanel. Don't add any game logic yet — just get the window to open."
Every vague word in a prompt is a decision the AI makes without your input.
Read both prompts aloud. Ask students to count the decisions the strong prompt makes for the AI: library (Swing), filename (SnakeGame.java), window size (600x600), title ('Snake'), class structure (JPanel subclass named GamePanel), scope (no game logic yet). That's 6 decisions transferred from AI to human. Six fewer surprises.

Anatomy of a Strong Prompt


🏷️ Context: "I'm building a Snake game in Java using Swing..." — tells the AI which ecosystem you're working in
📄 File name: "Create a single file called SnakeGame.java" — prevents the AI from splitting into files you didn't ask for
📐 Exact specs: "600×600 pixels", "title 'Snake'" — precise values, not vague descriptions
🧱 Class structure: "a JPanel subclass called GamePanel" — naming and design specified up front
🔭 Scope boundary: "Don't add any game logic yet" — scope limits = fewer surprises; iterating in small steps is safer
The scope boundary is the piece most beginners miss. When you leave scope open, the AI adds things you didn't ask for and don't understand. Iterating in small steps — get a window, then add the grid, then add movement — gives you a working checkpoint after each step.

When the AI Is Wrong

This will happen. It's expected. It's part of the workflow.


Step 1: Copy the error from the VS Code terminal (the red text)
Step 2: Paste it back into Copilot Chat or Claude
Step 3: Add context: "This happened when I tried to add the food logic"
Step 4: Read the AI's fix before accepting it — does the explanation make sense?

The AI can be confidently wrong. When something seems off, ask it to explain: "Why did you use a LinkedList here instead of an ArrayList?"
Optional live demo: introduce a deliberate compile error, show it in the terminal, paste it into the AI, get a fix. Normalize this loop. Students who think errors mean failure will disengage. Students who understand errors as input to the next prompt will keep going.

Snake Deep Dive


How the game works under the hood before you build it

Snake Architecture in One Screen

graph TD A["Timer tick (every 120-150 ms)"] --> B["Read pending direction"] B --> C["Compute new head cell"] C --> D{"Collision?"} D -- "Wall/self" --> E["Game over"] D -- "No" --> F{"Food?"} F -- "Yes" --> G["Grow snake + score++ + spawn food"] F -- "No" --> H["Move forward: add head, remove tail"] G --> I["Repaint panel"] H --> I I --> A style A fill:#1e3a5f,stroke:#4299e1,color:#fff style E fill:#2d1010,stroke:#ef5350,color:#fff style G fill:#1a3a2e,stroke:#48c78e,color:#fff
Students should understand the update loop before writing code. This mental model helps them debug.

Data Structures and Why They Fit

Game conceptJava typeWhy this choice
Snake bodyLinkedList<Point>Fast add/remove at head and tail each tick
Directionenum Direction {UP, DOWN, LEFT, RIGHT}Prevents invalid string values and typo bugs
Food positionPointSimple x,y coordinate for one pellet
Game stateboolean running, int scoreClear control over loop and UI messaging
Random spawnRandomUniform random food placement on grid
Strong prompt tip: ask the AI to explain each data structure choice in one sentence.

Incremental Build Plan for Week 1

1. Window only: JFrame + GamePanel, no logic
2. Grid and snake rendering: fixed initial snake, paint squares
3. Timer movement loop: move head forward every tick
4. Input handling: arrow keys update direction safely
5. Food spawn and growth: score increments on eat
6. Collision and game over: wall + self checks
7. Polish extension: speed curve, restart, theme, or high score
Commit after every stage. Each stage should be runnable before moving on.
